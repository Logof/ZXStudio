
## 1. Реалии технологического стека (Rust + egui)
Выбранный стек накладывает определенные условия на разработку, которые необходимо учитывать для обеспечения кроссплатформенности:

* Аппаратное ускорение (wgpu / OpenGL): Интерфейс egui рендерится силами видеокарты (GPU) через библиотеку wgpu.
* Реалия: На старых ПК или Linux-системах без установленных проприетарных драйверов приложение может упасть. Архитектура инициализации в main.rs должна предусматривать автоматический откат (fallback) на программный рендеринг glow (OpenGL ES) при сбое Vulkan/DirectX.
* Кроссплатформенность компиляторов (Проблема bin/): Движки компиляции Си (zcc.exe, msc.exe) — это нативные скомпилированные бинарники. Они не могут работать на macOS или Linux "из коробки".
* Реалия: Внутри каталога bin/ организуется строгая изоляция по операционным системам (bin/windows/, bin/linux/, bin/macos/). Среда разработки на Rust определяет текущую ОС на этапе выполнения с помощью макроса cfg!(target_os) и динамически строит пути к утилитам.
* Отсутствие сборщика мусора (Garbage Collector): Время жизни текстур в видеопамяти (например, загруженный тайлсет work.png) должно строго контролироваться через структуру egui::TextureHandle. При закрытии проекта текстура должна уничтожаться принудительно, чтобы не вызывать утечек VRAM.

## 2. Нефункциональные требования к разработке (Quality Gates)## Производительность и легкость

* Потребление памяти (RAM): В режиме покоя — не более 40 МБ. При пиковой нагрузке (рендеринг карты 16x16 экранов) — не более 100 МБ.
* Скорость запуска: Время от клика по иконке до отображения Мастера проектов — менее 0.5 секунд.
* Отрисовка (FPS): Стабильные 60 кадров в секунду на холсте карты при зуме и скроллинге. Достигается за счет использования egui::Shape для пакетного (batched) рендеринга сетки за один вызов отрисовки (draw call).

## Надежность и безопасность данных

* Изоляция сбоев (Panic-free): Любая дисковая операция (чтение поврежденного .png, сохранение на защищенный от записи диск) обязана обрабатываться через сопоставление шаблонов match или оператор ?. Паника приложения (crash) при ошибках ввода-вывода запрещена.
* Теневое резервирование (Autosave): Каждые 5 минут IDE должна в фоновом потоке сохранять резервную копию JSON-матрицы во временный файл map/.mapa.prj.tmp, предотвращая потерю прогресса пользователя при выключении питания.

## 3. Финальная структура производственного проекта и дистрибутива
Для сборки итогового .tap файла силами встроенного компилятора, структура каталогов на диске разработчика должна объединять код самой IDE (на Rust) и изолированное дерево игрового движка (на Си/Ассемблере):

zx_project_root/             # КОРНЕВАЯ ДИРЕКТОРИЯ РАЗРАБОТКИ
├── Cargo.toml               # Конфигурация сборки IDE на Rust
│
├── src/                     # ИСХОДНЫЙ КОД IDE (Rust + egui)
│   ├── main.rs              # Точка входа приложения
│   ├── app.rs               # Диспетчер интерфейса и оконных систем
│   ├── models/              # Модели данных (project, screen, config)
│   ├── ui/                  # Компоненты UI (map_editor, script_ide, configurator)
│   └── core/                # Ядро (io, validator, compiler_runner)
│
├── templates/               # ШАБЛОНЫ ГЕНЕРАЦИИ КОДА
│   ├── config.h.template    # Заготовка физики и баланса движка
│   └── memory_map.template  # Заготовка конфигурации памяти 48K/128K
│
├── bin/                     # КРОССПЛАТФОРМЕННЫЕ КОМПИЛЯТОРЫ И УТИЛИТЫ
│   ├── windows/             # Набор для Windows (.exe)
│   │   ├── zcc.exe          # Компилятор Z88DK (Си -> Z80)
│   │   └── msc.exe          # Компилятор скриптов .spt
│   ├── linux/               # Набор для Linux (ELF бинарники)
│   └── macos/               # Набор для macOS (Mach-O бинарники)
│
├── engine/                  # СТАТИЧНОЕ СИ-ЯДРО ИГРОВОГО ДВИЖКА (Прошивается в TAP)
│   ├── churromain.c         # Главный Си-файл игры, принимающий автогенерацию
│   ├── decompressor.asm     # Ассемблерный распаковщик экранов
│   └── player.c             # Логика перемещения персонажа
│
├── gfx/                     # ВХОДЯЩИЕ ГРАФИЧЕСКИЕ АССЕТЫ (PNG)
│   ├── work.png             # Исходный тайлсет
│   ├── sprites.png          # Спрайтсет с масками
│   └── title.png            # Экран заставки (для валидатора)
│
├── map/                     # РЕЗУЛЬТАТЫ РАБОТЫ РЕДАКТОРА КАРТ
│   └── mapa.prj             # Единый проектный файл JSON
│
├── dev/                     # ДИРЕКТОРИЯ АВТОГЕНЕРАЦИИ (Сюда пишет Rust ядро)
│   ├── config.h             # Сгенерированный Си-конфиг
│   └── memory_map.h         # Сгенерированная конфигурация банков памяти
│
└── output/                  # КОНЕЧНЫЙ РЕЗУЛЬТАТ СБОРКИ
    └── game.tap             # Скомпилированный образ игры для эмулятора

## 4. Спецификация автоматизированной сборки (Связующий слой)
Когда пользователь нажимает в интерфейсе кнопку 🚀 Компиляция ресурсов, бэкенд на Rust в модуле compiler_runner.rs выполняет последовательную цепочку атомарных системных вызовов:

   1. Парсинг и Шаблонизация: Ядро считывает текущее состояние памяти IDE, открывает файлы из templates/, производит замену маркеров и сохраняет готовые config.h и memory_map.h в папку dev/.
   2. Компиляция кода: Через std::process::Command вызывается нативная версия zcc из папки bin/ соответствующей ОС. Команда компиляции имеет вид:
   
   ./bin/[os]/zcc +zx -vn engine/churromain.c -Idev -o output/game.bin
   
   Флаг -Idev принудительно заставляет компилятор Си использовать сгенерированные файлы конфигурации из нашей папки автогенерации.
   3. Контейнеризация: Вызывается утилита сборки контейнера, которая упаковывает скомпилированный бинарный файл game.bin в финальный образ output/game.tap, добавляя туда стандартный загрузочный BASIC-блок.

Данная структура является полностью автономной, переносимой и гарантирует сборку финального .tap файла без необходимости установки в операционную систему пользователя сторонних компиляторов и сред выполнения.
