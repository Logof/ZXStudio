На основе всестороннего анализа ограничений и архитектуры движка La Churrera (MTE MK1) от Mojon Twins, а также трех частей присланного вами реального файла конфигурации, подготовлен полный, сквозной Roadmap реализации промышленной версии IDE на стеке Rust + egui.
Этот документ служит финальным техническим заданием (Blueprint) для полной сборки и закрытия всех этапов разработки проекта [2].

## 🗺️ Полный производственный Roadmap: ZX Spectrum Core IDE## 🛠️ ЭТАП 1. РАСШИРЕНИЕ МОДЕЛЕЙ ДАННЫХ (Data Layout)
Цель: Обеспечить 100% совместимость JSON-модели ProjectData со всеми физическими, экранными и логическими макросами из присланного config.h.

* 1.1. Внедрение параметров HUD (src/models/config.rs):
* Создать переменные координат знакомест: viewport_x (0), viewport_y (2), life_x (30), life_y (8), objects_x (30), objects_y (12), keys_x (30), keys_y (16), game_over_attr (15).
* 1.2. Внедрение Fixed-Point физики движения (src/models/config.rs):
* Добавить целочисленные поля хранения констант: player_max_vx (256), player_ax (32), player_rx (24).
   * Добавить параметры констант Side-View движка (для обратной совместимости при переключении в Мастере): player_max_vy_cayendo (512), player_g (32), player_vy_inicial_salto (64).
* 1.3. Внедрение массива физического поведения тайлов (src/models/screen.rs):
* Добавить в модель ProjectData фиксированный вектор tile_behaviours: Vec<u8> размером строго 32 байта для хранения физических масок (0 — Walkable, 1 — Kills, 8 — Wall obstacle) первых двух строк палитры. Инициализировать его дефолтным массивом из присланного Си-кода.

## 🎨 ЭТАП 2. МОДЕРНИЗАЦИЯ ИНТЕРФЕЙСА (UX/UI Windows Configuration)
Цель: Реализовать интерактивное управление новыми параметрами внутри модулей нашей разделенной Docking-системы.

* 2.1. Создание визуального HUD-инспектора (src/ui/configurator.rs):
* Отрендерить в правой панели интерактивную сетку знакомест ZX Spectrum размером 32x24 блока (каждое знакоместо 8x8 пикселей, масштабированное на экране разработчика).
   * Выделить серым цветом область игрового окна (Viewport) на основе значений VIEWPORT_X и VIEWPORT_Y.
   * Реализовать перетаскивание (Drag-and-Drop) мышью текстовых меток [LIFE], [OBJECTS], [KEYS] строго по ячейкам знакомест с автоматической перезаписью координат в модель. Внедрить встроенную проверку: если метка заходит внутрь Viewport — окрашивать её в неоновый красный цвет и выводить предупреждение.
* 2.2. Конструктор физической матрицы тайлов (src/app/tab_viewer.rs):
* Модифицировать золотистый инспектор свойств во вкладке CustomTab::TilePalette.
   * При левом клике на любой из 32 тайлов палитры выводить выпадающий список (ComboBox) выбора его физической роли ( Walkable, 💀 Kills, 🧱 Full Obstacle). Изменение значения мгновенно перезаписывает байт в tile_behaviours по индексу тайла.

## 🔄 ЭТАП 3. ИНТЕГРАЦИЯ ПРАВИЛ ДВИЖКА НА ХОЛСТ И ПАЛИТРУ
Цель: Привязать механики присланного config.h к интерактивному 3-слойному холсту 15x10 тайлов.

* 3.1. Управление логикой Legacy Refills (src/app/tab_viewer.rs):
* Присланная директива #define LEGACY_REFILLS означает, что аптечки спавнятся в коде игры на месте взятых предметов.
   * Действие: В режиме «Хотспоты» палитры заблокировать выбор и установку Аптечки (Тип 3), выводить её серым цветом и показывать всплывающую подсказку: «Включен Legacy-режим: рефилы здоровья будут генерироваться Си-движком автоматически на месте квестового лута. Ручное размещение на карте не требуется».
* 3.2. Обработка флага NO_MAX_ENEMS:
* Поскольку присланный Си-движок Churrera официально поддерживает пустые слоты врагов на экранах, холст карты избавляется от необходимости заполнения пустых слотов фейковыми объектами. Если на экране стоит 1 или 2 врага, Rust-код пишет в JSON только их, а пустые слоты инициализирует типом 0 (отсутствует).

## ⚙️ ЭТАП 4. ДВИЖОК ШАБЛОНИЗАЦИИ И ЭКСПОРТА (Template Engine Engine)
Цель: Переписать генератор src/core/io.rs для формирования точной копии присланного файла.

* 4.1. Обновление текстового шаблона: Заменить контент templates/config.h.template полной копией вашего реального файла, заменив хардкодные цифры и закомментированные строки на маркеры подстановки {{}}.
* 4.2. Алгоритм строкового маппинга (src/core/io.rs):
* Реализовать чистые функции сборки Си-строк. Константы скоростей (например, PLAYER_MAX_VX) берутся из слайдеров интерфейса и прописываются как целые числа.
   * Массив поведения тайлов разворачивается в легитимную Си-строку:
   
   let mut beh_string = String::from("unsigned char comportamiento_tiles [] = {\n\t");for (i, beh) in project.tile_behaviours.iter().enumerate() {
       beh_string.push_str(&format!("{}, ", beh));
       if (i + 1) % 16 == 0 { beh_string.push_str("\n\t"); }
   }
   beh_string.push_str("\n};");
   
   * Все строки подставляются в шаблон методом .replace() и записываются в целевую папку компилятора dev/config.h.

## 🎯 ЭТАП 5. ВЫДЕЛЕНИЕ ПАМЯТИ (RAM AUDIT) И ВНЕШНИЕ КОМПИЛЯТОРЫ
Цель: Автоматизировать расчет пула спрайтов, сборку бинарных файлов ресурсов мира и кроссплатформенный вызов сборочного инструментария.

* 5.1. Автоматический расчет NUMBLOCKS (src/app/mod.rs):
* На основе активного режима PLAYER_MOGGY_STYLE (вид сверху) и закомментированного #define PLAYER_CAN_FIRE (стрельба отключена), бэкенд на Rust вычисляет точный размер графического буфера в ОЗУ Спектрума.
   * Формула: Базовые_кадры_персонажа_TopView (16 кадров) * 10 блоков памяти. Так как пули отключены, прибавка MAX_BULLETS * 5 автоматически сбрасывается в 0, высвобождая память под Си-код хуков. Результат перезаписывается в churromain.c.
* 5.2. Модуль трансляции compiler_runner.rs:
* Создать файл src/core/compiler_runner.rs. Внедрить в него кроссплатформенный запуск компилятора Z88DK с помощью std::process::Command. Срипт будет определять операционную систему (windows, linux, macos) [2] и запускать нативную версию zcc из внутренней папки проекта bin/.
   * Выходной артефакт: Утилита связывает бинарник карты mapa.map, структуры врагов enems.ene, сгенерированный Си-конфиг config.h и упаковывает всё в готовый к запуску контейнер output/game.tap.
